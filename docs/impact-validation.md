# Impact validation status

TraceForge currently proves a technical claim in a controlled returns-workflow laboratory. It does **not** yet claim customer adoption, production savings, or independent maintainer endorsement.

## Evidence already available

- A reproducible source run with model provenance, failed proofs, deterministic assertions, and a recomputable final digest.
- A separately exercised Local Runner path from a fresh pinned tag clone.
- Measured technical results for the bounded corpus: seven scenarios, 35 assertions, zero mismatches after repair.

These are implementation and correctness signals. They are not user-impact evidence.

## Independent validation still required

Before making any impact claim in the submission, obtain at least one of the following:

1. A legacy-system maintainer completes the hosted replay, opens the proof bundle, and records what was understandable or missing.
2. A timed comparison measures the same bounded migration task with and without TraceForge, including setup and human review time.
3. A second unfamiliar Web + REST + SQLite/PostgreSQL workflow is migrated without project-specific hard-coding.

## Maintainer session protocol

Record consent before attributing a name, employer, quote, or screenshot.

1. Start at the hosted landing page without coaching.
2. Measure time until the reviewer can state the inferred high-value rule.
3. Ask the reviewer to identify the intentionally failed candidate and the subsequent repair.
4. Ask the reviewer to recompute or inspect the proof digest.
5. Record confusion, manual interventions, elapsed time, and whether the proof would be sufficient for a migration review.
6. Preserve raw notes privately; publish only approved, attributable statements.

## Submission boundary

Until one of the independent validations above is complete, submission copy should say **potential impact** and describe the target problem. It must not say “validated by maintainers,” quote time savings, or imply production use.
